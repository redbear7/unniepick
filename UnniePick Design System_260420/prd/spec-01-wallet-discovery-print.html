01

문제 정의
& 가설

저장된 쿠폰의 대부분은 '만료되고 나서' 존재가 기억된다. 앱을 열지 않은 순간이 가장 중요한 접점이다.

Problem
"쿠폰 저장해놓고
까먹어요."
언니픽 사용자의 저장된 쿠폰 중 약 58%가 사용 없이 만료된다 (내부 가정치). 사용자는 가게 앞을 지나가면서도 쿠폰의 존재를 기억하지 못한다. 쿠폰은 '저장하는 순간'이 아니라 '결제 직전'에 가치가 발생한다.
Hypothesis
"위치 기반 능동 알림이
사용률을 2배로."
IF 저장된 쿠폰이 있는 매장 반경 100m에 진입할 때 Live Activity·Widget으로 자동 리마인드한다면, THEN 쿠폰 사용률이 40% → 65%로 상승하고 W1 리텐션이 +8%pt 개선된다.
Why now

iOS 16+ Live Activity·Dynamic Island API가 안정화되면서 잠금화면 접근 비용이 낮아졌다. 경쟁 서비스 중 로컬 커머스 카테고리에서 Live Activity를 전면 채택한 곳은 아직 없다 — 선점 기회.

03

UX 플로우 ·
Step by Step

백그라운드에서 위치 변경 감지 → 후보 쿠폰 매칭 → Live Activity 시작 → 탭하면 매장 상세로.

1
Background위치 변화 감지
기기가 유의미한 위치 변화(약 500m)를 감지하면 OS가 언니픽 앱을 잠깐 깨운다. 실시간 GPS가 아니라서 배터리 소모 거의 없음.
CLLocationManager.startMonitoringSignificantLocationChanges()
2
Background후보 쿠폰 매칭
현재 위치에서 500m 반경 내 매장 중, 사용자 지갑에 저장된 active 쿠폰이 있고 만료가 72시간 이내인 케이스를 쿼리.
GET /v1/wallet/nearby?lat=..&lng=..&radius=500
3
ForegroundLive Activity 시작
매칭된 쿠폰이 있으면 Live Activity를 시작. 잠금화면과 Dynamic Island에 "쿠폰 1장 · 파니스 · 18:00 만료" 표시. 알림음 없음, 배지만.
ActivityKit.Activity<CouponAttributes>.request()
4
User탭 → 매장 상세
사용자가 Live Activity를 탭하면 앱이 열리고 해당 쿠폰의 매장 상세 화면으로 딥링크. 쿠폰 사용 CTA 1탭 거리 (§04 쿠폰 사용 UX 참조).
unniepick://coupon/{coupon_id}
5
System자동 종료
사용자가 반경에서 벗어나거나, 쿠폰을 사용하거나, 8시간이 지나면 Live Activity 종료. 스팸 방지.
activity.end(.dismissalPolicy(.immediate))
Throttling rules
  • 동일 매장당 최대 1회/일 알림
  • 전체 합산 하루 최대 3회
  • 사용자가 Live Activity를 dismiss하면 해당 쿠폰은 24시간 쿨다운
  • 22:00 – 09:00 quiet hours는 Widget만 업데이트, Live Activity 미시작
04

쿠폰 사용 UX
· 터치형 & 캡쳐 방지

QR 스캔 대신, 사용자가 쿠폰 카드를 길게 눌러 현장 토큰을 생성하고 사장님이 눈으로 확인하는 방식. 캡쳐·공유·재사용을 기술적으로 차단한다. 이 UX는 Phase 1 3개 아이디어 모두에서 공용으로 사용된다.

Why not QR
QR은
동네 가게에 맞지 않는다.
소상공인은 고정 QR 스탠드를 새로 두고, 네트워크 문제나 카메라 초점이 안 맞으면 줄이 밀린다. "손님 앞에서 헤매면 안 된다"는 게 사장님들 1순위 요구. 게다가 QR은 화면만 보여주면 되는 '인증'이 아니라 '스캔'이 필요해 양쪽 다 번거롭다.
Touch-to-reveal
꾹 누르면 열린다.
사장님이 눈으로 본다.
사용자가 쿠폰 카드를 1.5초 길게 누르면 전체 화면 '사용 모드' 진입 → 빨간 사용 CTA + 실시간 시각 애니메이션 + 6자리 현장 토큰. 사장님은 숫자·애니메이션만 눈으로 확인 → "사용완료" 탭. 스캔도 없고 추가 하드웨어도 없다.
Anti-Capture · 3중 방어

캡쳐·공유·재사용을 어떻게 막는가

🛡 Layer 1
동적 토큰
(시각 혼합 시리얼)

쿠폰 일련번호와 현재 시각(초 단위)을 HMAC-SHA256으로 섞어 10초마다 갱신되는 6자리 코드 생성. 10초 지난 토큰은 서버에서 거부. 스크린샷은 도착 전에 이미 만료.

🛡 Layer 2
FLAG_SECURE
(스크린샷 차단)

사용 모드 진입 시 iOS UIScreen.captured 감지 → 화면 블러. Android는 FLAG_SECURE로 OS 레벨 캡쳐 자체 불가. 녹화·미러링도 차단.

🛡 Layer 3
라이브 애니메이션
(시각 서명)

쿠폰 배경의 브랜드 그라디언트·파동 애니메이션이 매초 움직인다. 사장님 체크리스트의 1번은 "움직이는지". 정지 이미지는 무효.

Token 생성 규칙

서버·기기 공통 HMAC 알고리즘

// 기기·서버 양쪽에서 동일 계산 function redeemToken(couponId, userId, now) { const window = Math.floor(now / 10000); // 10초 윈도우 const message = `${couponId}:${userId}:${window}`; const hmac = crypto.hmac( 'SHA256', SECRET_KEY, message ); // 6자리 숫자로 축약 (100만 개 공간) return parseInt( hmac.slice(0, 8), 16 ) % 1000000; }

서버는 현재 윈도우 ± 1개 (이전·현재·다음 10초) 까지만 유효 — 시계 오차 ±10초 허용, 그 이상은 거부.

재사용 차단

POST /v1/coupons/redeem

// 사장님 앱 → 서버 (사용완료 버튼) POST /v1/coupons/redeem { coupon_id: "cpn_7h2k...", redeem_token: 429183, user_device_hash: "sha256...", merchant_id: "mrc_paani_...", client_ts: 1730123456 } // 서버 검증 순서 // 1. 토큰이 해당 쿠폰·유저 조합으로 // 현재 10초 윈도우에 유효한가 // 2. coupon.status = 'active' 인가 // 3. redeemed_at IS NULL 인가 (멱등) // 4. 매장 GPS 반경 100m 이내인가 // → 통과 시 status='redeemed' UPDATE // 동시 요청은 SELECT FOR UPDATE 잠금
A
사용자 앱쿠폰 카드 롱프레스 1.5초
지갑에서 쿠폰 카드를 길게 누르면 햅틱 + 카드가 풀스크린으로 확장. 실수 탭 방지를 위해 1.5초 홀드.
onLongPressStart → Haptic.medium · expand 300ms
B
사용자 앱풀스크린 '사용 모드'
화면 전체가 쿠폰. 중앙에 6자리 현장 토큰 + 10초 카운트다운 링. 배경엔 브랜드 그라디언트 애니메이션. FLAG_SECURE ON.
activityType=RedeemSession · captureGuard.start()
C
사용자 → 사장님화면만 보여준다
사용자가 카운터에 폰을 건네준다. 사장님은 (1) 토큰 숫자, (2) 애니메이션이 살아있는지 둘 다 확인. 0.5초면 끝.
시각 체크만 · 터치 없음
D
사장님 앱"사용완료" 탭
사장님 앱 홈에서 "쿠폰 확인" 버튼 → 현재 매장 근처 대기 중인 쿠폰 리스트 → 해당 쿠폰 "사용완료". 서버 검증 후 즉시 반영.
POST /v1/coupons/redeem · 매장 GPS 반경 내 한정
E
사용자 앱즉시 '사용 완료' 전환
서버 push로 Live Activity와 풀스크린 모드 즉시 종료 → '사용 완료' 리포트 카드. 다른 기기·세션에서도 동기화.

New Table · 쿠폰 사용 UX용 (§04)

redeem_token_log — 재사용·충돌 차단

-- 토큰 사용 이력 · 멱등성 보장 CREATE TABLE redeem_token_log ( id BIGSERIAL PRIMARY KEY, coupon_id BIGINT NOT NULL, user_id BIGINT NOT NULL, token_window BIGINT NOT NULL, -- floor(epoch_ms / 10000) redeem_token INT NOT NULL, merchant_id BIGINT NOT NULL, verified_at TIMESTAMPTZ DEFAULT now(), gps_distance_m INT, status VARCHAR(16) -- 'success' | 'expired' | 'reused' -- | 'store_mismatch' | 'rejected' ); CREATE UNIQUE INDEX ux_rtl_coupon_success ON redeem_token_log(coupon_id) WHERE status = 'success'; -- 동일 쿠폰 2번 성공 불가 (멱등)
왜 이 테이블이 필요한가
  • 재사용 방지 · UNIQUE INDEX WHERE status='success' 로 DB 레벨에서 동일 쿠폰 재사용 자체가 불가능.
  • 공격 탐지 · 만료된 토큰·다른 매장 시도는 expired / store_mismatch 로 기록 → 이상치 모니터링.
  • 사장님 분쟁 대응 · "정말 썼는지" 이력 조회 가능. verified_at + gps_distance_m 으로 증거.
  • 감사 로그 · 캡쳐 방지 로직이 작동 안 된 케이스 역추적.
Retention

성공 건은 영구. 실패 건은 30일 후 파티션 DROP.

07

API 엔드포인트
제안

엣지 케이스
/ 실패 시나리오

⚠
Edge 01
위치 권한을 "앱 사용 중"만 허용한 경우
Significant Location Change는 Always 권한이 필요. Always 요청 프라이머 화면을 먼저 노출 → 거부 시 "앱 사용 중에만 근처 쿠폰 배너"로 다운그레이드.
대응
권한 상태를 이벤트로 기록 → Always 전환율 지표화. 첫 주 이후 리마인드 1회 허용.
⚠
Edge 02
사용자가 여러 매장이 몰린 거리(상가)에 있을 때
반경 500m 내 쿠폰 5장 이상 매칭 → Live Activity 1개에 "근처 쿠폰 5장" 요약 표시. 탭 시 전용 리스트 페이지.
대응
단일 > 그룹 결정 기준: 3장 이상이면 그룹. 만료 임박(6시간 이내)은 항상 단일 우선.
⚠
Edge 03
쿠폰을 이미 다른 기기에서 사용함
APNs Live Activity update로 즉시 종료. 사용자가 탭했을 때 이미 사용된 쿠폰을 보는 일 방지.
대응
쿠폰 redeem webhook → Activity end push 발송. 서버가 단일 소스.
⚠
Edge 04
Dynamic Island 없는 기기 (iPhone 14 이하 일반)
잠금화면·홈 위젯만 활용. 앱 사용 중일 때는 상단 배너로 대체.
대응
타겟 기기 iPhone Pro 라인 우선 → 14일 실험. 일반 모델은 Widget 중심으로 가치 측정.
⚠
Edge 05
지하철 탑승 중 수많은 매장을 빠르게 통과
속도 감지 (>30km/h) → 자동 발견 일시 중지. 하차 후 5분 + 도보 속도 확인되면 재개.
대응
CLActivityType.automotiveNavigation 감지. 모호한 경우는 발송 보류가 디폴트.

리스크
& 대응

!
Risk 01 · Privacy
"앱이 내 위치를 항상 추적한다"는 인식
Always 권한 요청이 iOS에서 가장 강한 경고 문구를 띄움. 사용자가 거부하면 기능 자체가 반쪽.
대응
(1) Significant Location 실시간 아님을 프라이머에서 강조 (2) "서버에 위치 저장 안 함 · 매칭만 사용" 약속 + 실제로 저장 안 함 (3) 언제든 OFF 1탭. 온보딩 중엔 물어보지 않음, 첫 쿠폰 저장 후 자연스럽게.
!
Risk 02 · Accuracy
도심 GPS 오차 50-100m
반경 100m로 설정하면 실제로는 가게 바로 앞에 있어도 놓치고, 반대로 옆 건물에서 뜬다.
대응
기본 반경 500m에서 시작 → 실사용 데이터로 조정. 빌딩 내부 보정은 Phase 2에서 iBeacon 검토.
!
Risk 03 · Battery
배터리 소모 우려로 초기 부정 리뷰
사용자가 느끼지 못해도 "위치 추적"이라는 단어에 대한 반감.
대응
SLC는 셀타워 기반이라 실제 배터리 영향 <1%. App Store 상세·온보딩에 배터리 임팩트 정량 명시.
!
Risk 04 · Spam 체감
3회 상한 넘지 않아도 "자주 뜬다" 인식
알림 피로는 빈도보다 타이밍 미스에서 생긴다. "그 시간에 왜 뜸?"이 반복되면 OFF.
대응
dismiss 쿨다운 24시간 엄수 · 식사 시간대(11:30-13:30, 18:00-20:00) 가중치 · 방해금지 시간 기본 22-9시.
!
Risk 05 · Store Coverage
매장 좌표 데이터 품질 편차
기존 매장 등록 시 좌표가 도로명 주소 geocoding으로만 들어가 있어 실제 위치와 50m 이상 차이나는 경우.
대응
사장님 앱에 "내 가게 위치 조정" 지도 핀 기능 추가. 초기 파일럿 매장만 수동 검수.

오픈 퀘스천

스펙 확정 전 PM·디자이너·엔지니어가 함께 답해야 할 질문들.

?
Q1 · Product
Always 권한 요청 타이밍 — 온보딩 vs. 첫 쿠폰 저장 후?
온보딩 중이면 허용률 높지만 거부 시 영구 반려 가능성. 저장 후면 맥락이 명확하지만 허용률 낮을 수 있음.
결정 기준
초기엔 B안(맥락 중시). 4주 후 허용률 데이터 보고 A/B로 재검증.
?
Q2 · Design
Live Activity에 CTA 버튼을 둘 것인가?
"QR 바로 열기" 버튼을 두면 전환율은 오르지만 실수 탭 가능성. 탭 전체를 앱 딥링크로 두는 게 안전.
결정 기준
MVP는 탭 전체만. GA 후 실수 탭율 <3% 확인되면 버튼 추가 실험.
?
Q3 · Business
매장 측에서 "자동 발견 대상" 토글을 돈 받고 팔 것인가?
Meituan 사례 학습 — 노출 보장 프리미엄이 수익 모델. 다만 초기엔 전 매장 기본 ON으로 사용자 가치 검증이 먼저.
결정 기준
Phase 1은 전 매장 ON. Phase 2부터 "프리미엄 반경 1km" 유료 옵션 검토.
?
Q4 · Technical
iOS 16.1 미만 사용자 (약 10%) 어떻게 할 것인가?
Live Activity 미지원. 푸시 알림으로 다운그레이드 or 기능 미제공.
결정 기준
로컬 푸시로 다운그레이드 (동일 엔드포인트 재사용). Widget은 iOS 14+라 대부분 커버.
