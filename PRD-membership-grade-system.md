# PRD — 언니픽 회원등급제 & 가격제보 보상 시스템

**버전** 1.0 | **작성일** 2026-04-22 | **상태** 초안

---

## 1. 요약

언니픽 회원을 10단계 등급으로 세분화하고, 가격 제보·투표 등 앱 내 기여 행동에 포인트 보상을 연결한다.
사용자가 더 오래, 더 자주 앱을 쓸 이유를 만들고, 가격 정보 품질을 자연스럽게 높인다.

## 3. 배경

### 현재 상황
- 등급: 브론즈 / 실버 / 골드 3단계 — 구분이 너무 단순해 동기부여 약함
- 가격 제보 기능 신규 출시 — 보상 체계 없으면 제보 지속률 낮아질 가능성
- 누적 포인트(`profiles.level_progress`)는 DB에 있지만 사용자에게 의미가 없음

### 왜 지금인가
가격 제보 기능이 막 출시됐다. 초기 사용자 경험이 곧 제보 문화를 만든다.
지금 보상 체계를 붙이지 않으면 나중에 바꾸기 어렵다.

---

## 4. 목표

### 목적
> "앱을 쓸수록 나의 레벨이 올라간다"는 성취감을 주어, 재방문·기여 행동을 늘린다.

### 회사에 주는 이익
- 사용자당 쿠폰 사용 횟수 증가 → 가맹점 매출 기여 → 수수료 수익
- 가격 제보 데이터 축적 → 앱 고유 콘텐츠 강화 → 경쟁 앱 차별화

### 핵심 지표 (OKR 형식)

| 지표 | 현재 | 목표 (출시 후 3개월) |
|------|------|---------------------|
| 월간 가격 제보 수 | 0 | 500건 이상 |
| 7일 재방문율 | 측정 시작 | 40% 이상 |
| 레벨 3 이상 도달 비율 | - | 신규 가입자의 30% |
| 쿠폰 사용 1인 평균 | 측정 시작 | 월 2회 이상 |

## 5. 타겟 사용자

### 주 타겟
- **창원 거주 20~40대 여성**
- 카페·맛집·뷰티샵을 자주 이용하며 가격에 민감
- 앱으로 할인·혜택 챙기는 걸 즐김

### 사용자 유형별 니즈

| 유형 | 니즈 | 이 시스템이 주는 것 |
|------|------|-------------------|
| 절약러 | 더 많은 할인 | 레벨업 시 전용 쿠폰 |
| 탐방러 | 새 가게 발견 | 탐색 포인트 적립 |
| 기여러 | 인정받고 싶음 | 배지·랭킹 노출 |
| 캐주얼 | 귀찮은 거 싫음 | 자동 적립, 낮은 진입장벽 |

## 6. 가치 제안

### 사용자가 얻는 것
- 행동할 때마다 포인트가 쌓이는 **즉각적인 보람**
- 레벨이 올라갈수록 더 좋은 혜택을 받는 **계층적 보상**
- 내 제보가 지도에 반영되는 **실질적인 기여감**

### 사용자가 피할 수 있는 것
- "왜 앱을 써야 하지?"라는 의문
- 가격 정보를 직접 찾아다녀야 하는 번거로움

### 경쟁 대비 차별점
- 단순 쿠폰 앱들은 등급제 없음
- 가격 제보 + 투표 + 보상이 연결된 구조는 창원 로컬 앱 중 없음

### 7.1 포인트 적립 체계

모든 포인트는 `profiles.total_points`에 누적되며 등급 계산에 사용된다.

| 행동 | 포인트 | 제한 |
|------|--------|------|
| 앱 첫 가입 | +100P | 1회 |
| 매일 앱 방문 | +2P | 1일 1회 |
| 쿠폰 다운로드 | +5P | 1일 10회 |
| 쿠폰 사용 완료 | +30P | 제한 없음 |
| 가게 팔로우 | +5P | 가게당 1회 |
| 스탬프 적립 | +5P | 1일 1회/가게 |
| 영수증 인증 | +20P | 1일 2회 |
| 친구 초대 (가입 완료 시) | +150P | 무제한 |
| **가격 제보 등록** | **+30P** | **1일 3회** |
| **맞아요 투표** | **+2P** | **1일 20회** |
| **틀려요 투표** | **+2P** | **1일 20회** |
| **내 제보 맞아요 10개 달성** | **+20P** | **제보당 1회** |
| **내 제보 맞아요 50개 달성** | **+50P + 쿠폰 1장** | **제보당 1회** |
| **내 제보가 대표 가격 채택** | **+100P + 특별 배지** | **제보당 1회** |

### 7.2 회원 등급 — 10단계

등급 이름은 언니픽의 감성(로컬·따뜻함·여성)을 담아 꽃 이름으로 정했다.

| 레벨 | 등급명 | 이모지 | 누적 포인트 | 특징 |
|------|--------|--------|------------|------|
| 1 | 새싹 | 🌱 | 0 ~ 199P | 시작 단계 |
| 2 | 꽃봉오리 | 🌸 | 200 ~ 499P | 앱에 익숙해지는 단계 |
| 3 | 들꽃 | 🌼 | 500 ~ 999P | 혜택을 알아가는 단계 |
| 4 | 장미 | 🌹 | 1,000 ~ 1,799P | 단골 사용자 진입 |
| 5 | 목련 | 🤍 | 1,800 ~ 2,999P | 앱 핵심 사용자 |
| 6 | 벚꽃 | 🌺 | 3,000 ~ 4,799P | 커뮤니티 기여자 |
| 7 | 작약 | 💐 | 4,800 ~ 7,199P | 앱 홍보대사급 |
| 8 | 난초 | 🪷 | 7,200 ~ 10,499P | 상위 10% |
| 9 | 매화 | ✨ | 10,500 ~ 14,999P | 상위 3% |
| 10 | 언니픽 퀸 | 👑 | 15,000P~ | 최고 등급, 상위 1% |

> **설계 원칙**  
> - 레벨 1→3: 쉽게 달성 (2주 내 가능) → 이탈 방지  
> - 레벨 4→6: 꾸준한 사용 필요 (1~3개월) → 습관 형성  
> - 레벨 7→10: 기여 행동 필수 (가격 제보, 초대) → 파워유저 육성

### 7.7 DB 스키마 변경

```sql
-- profiles 테이블 컬럼 추가
ALTER TABLE profiles
  ADD COLUMN IF NOT EXISTS total_points    INTEGER NOT NULL DEFAULT 0,
  ADD COLUMN IF NOT EXISTS level_number    INTEGER NOT NULL DEFAULT 1,  -- 1~10
  ADD COLUMN IF NOT EXISTS badge_ids       TEXT[]  DEFAULT '{}';        -- 배지 ID 배열

-- 포인트 내역 테이블 (신규)
CREATE TABLE point_history (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  action     TEXT NOT NULL,        -- 'price_report', 'coupon_use', 'vote' 등
  points     INTEGER NOT NULL,     -- 양수=적립, 음수=차감
  label      TEXT,                 -- "가격 제보 등록", "쿠폰 사용" 등 표시용
  ref_id     UUID,                 -- 관련 리소스 ID (제보 ID, 쿠폰 ID 등)
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 등급 계산 함수 (total_points → level_number)
CREATE OR REPLACE FUNCTION calc_level(points INTEGER) RETURNS INTEGER AS $$
BEGIN
  RETURN CASE
    WHEN points <   200 THEN 1
    WHEN points <   500 THEN 2
    WHEN points <  1000 THEN 3
    WHEN points <  1800 THEN 4
    WHEN points <  3000 THEN 5
    WHEN points <  4800 THEN 6
    WHEN points <  7200 THEN 7
    WHEN points < 10500 THEN 8
    WHEN points < 15000 THEN 9
    ELSE 10
  END;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- total_points 변경 시 level_number 자동 갱신 트리거
CREATE OR REPLACE FUNCTION sync_level_number()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.level_number := calc_level(NEW.total_points);
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_sync_level
  BEFORE UPDATE OF total_points ON profiles
  FOR EACH ROW EXECUTE FUNCTION sync_level_number();
```

### 7.8 가정 (검증 필요)

| 가정 | 검증 방법 |
|------|----------|
| 사용자가 포인트를 위해 가격 제보를 한다 | 출시 1개월 후 제보 수 확인 |

| 레벨 6+ 혜택(쿠폰)이 사업적으로 지속 가능하다 | 쿠폰 발행 비용 vs 재방문 매출 분석 |

## 8. 출시 계획

### Phase 1 — MVP (4주)
**목표:** 포인트 적립 + 10단계 레벨 + MY 탭 노출

- [ ] DB: `total_points`, `level_number`, `point_history` 추가
- [ ] 포인트 적립 함수 (`addPoints()`) 구현
- [ ] 가격 제보 시 +30P 자동 지급 연결
- [ ] 맞아요/틀려요 투표 시 +2P 자동 지급 연결
- [ ] MY 탭 ProfileHeader: 새 등급명·이모지·포인트 표시
- [ ] LevelProgress 바: 현재 레벨 → 다음 레벨 기준으로 재계산

### Phase 2 — 배지 + 보상 고도화 (2주)
**목표:** 배지 시스템 + 맞아요 달성 보상

- [ ] 배지 판정 로직 (트리거 or 앱 내 체크)
- [ ] MY 탭 배지 섹션 UI
- [ ] 맞아요 10개/50개 달성 감지 → 포인트 지급 + 푸시
- [ ] 포인트 내역 화면 (PointHistoryScreen)

### Phase 3 — 등급 혜택 연결 (4주)
**목표:** 레벨에 따른 쿠폰·알림 차등 제공

- [ ] 레벨별 전용 쿠폰 발행 (어드민 연동)
- [ ] 레벨업 시 푸시 알림
- [ ] VIP 배지 지도 표시 (레벨 8+)
- [ ] 레벨 랭킹 (옵션)

### 제외 항목 (Phase 3 이후)
- 포인트 → 현금 전환
- 가맹점이 직접 포인트 혜택 설정
- 친구 간 포인트 선물
